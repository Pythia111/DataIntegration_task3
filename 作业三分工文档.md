基于XML的异构教务数据集成系统
一、项目概述
本项目基于XML技术，实现学院A、B、C三个异构教务系统的数据集成。三个学院分别使用SQL Server、Oracle、MySQL数据库，拥有独立的学生、课程及选课信息。目标是通过新增集成服务器，实现跨学院课程共享、选课信息同步、全局统计以及退选课流程。每个学院需包含50名学生、10门课程，每名学生选修5门课。各学院系统需具备GUI及登录功能。
二、小组角色与总体分工
角色	成员	主要职责
项目经理 & 集成架构师	成员1	总体设计、进度管理、接口定义、文档与流程图绘制
学院A系统开发	成员2	学院A教务系统（SQL Server + GUI + 登录 + 本地业务）
学院B系统开发	成员3	学院B教务系统（Oracle + GUI + 登录 + 本地业务）
学院C系统开发	成员4	学院C教务系统（MySQL + GUI + 登录 + 本地业务）
集成服务器核心开发	成员5	集成服务器通信模块、XML验证与转换（XSLT）、课程共享与选课转发
集成服务器扩展功能	成员6	全局统计模块、退选课流程实现、系统联调与异常处理
三、详细任务分解
1. 成员1：项目经理 & 集成架构师
- 需求分析与设计：细化集成场景流程，设计系统整体架构（参考文档3.5.2架构图）。
- 接口定义：定义各学院与集成服务器之间的XML消息格式（学生、课程、选课、退选等），编写对应的XSD Schema文件（参考文档表3-12～3-15）。
- 流程图绘制：绘制课程共享、跨院选课、退选课、全局统计等核心业务流程图（用于报告）。
- 进度管理：制定迭代计划，协调成员间接口对接，组织代码审查。
- 文档撰写：汇总实验报告，撰写系统设计说明、集成方案、测试用例及结果分析。
交付物：
- 系统架构设计文档
- XML Schema定义文件（至少6个：统一学生、统一课程、统一选课、各学院转换映射所需）
- 业务流程图（不少于4张）
- 最终实验报告
1. 成员2：学院A系统开发（SQL Server）
- 数据库设计与实现：在SQL Server中创建学生表、课程表、选课表、账户表（参考文档表3-2～3-5）。插入50名学生、10门课程，并为每名学生生成5条选课记录。
- GUI开发：使用Swing/JavaFX或Python Tkinter实现学院A教务系统界面，包含：
  - 登录窗口（学生/管理员）
  - 学生端：查看本校课程、查看共享课程（从集成服务器获取）、跨院选课、退选课、查看个人课表
  - 管理员端：课程共享标记设置、本地课程管理
- 本地业务逻辑：处理本校学生选本校课程；对于跨院选课请求，调用XML生成模块发送至集成服务器；接收集成服务器返回的选课结果并更新本地选课表。
- XML生成与解析：使用DOM4J或JAXB，按照约定格式生成本院学生、课程信息的XML文件；解析集成服务器下发的XML数据并存入本地数据库。
- 通信模块：通过HTTP或Socket向集成服务器发送请求（如课程查询、选课、退选），并接收响应。
交付物：
- 学院A完整源代码（含数据库脚本、GUI代码、XML处理、通信代码）
- 数据库备份文件（含初始50名学生、10门课程、选课数据）
- 部署与运行说明
1. 成员3：学院B系统开发（Oracle）
- 数据库设计与实现：在Oracle中创建表结构（参考文档表3-6～3-9）。数据量与学院A一致：50名学生、10门课程、每生5门选课。
- GUI开发：与学院A类似的功能，注意界面风格可与学院A不同但需满足登录和选课需求。
- 本地业务逻辑：实现本校选课、跨院选课请求转发、共享课程设置等。
- XML处理：生成符合学院B本地格式的XML，并能解析集成服务器转换后的XML。
- 通信模块：与集成服务器交互，实现课程共享请求、选课请求、退选请求。
交付物：
- 学院B完整源代码（含Oracle建表语句、GUI、XML、通信模块）
- 初始数据导出文件（SQL*Loader或Dump文件）
- 运行手册
1. 成员4：学院C系统开发（MySQL）
- 数据库设计与实现：在MySQL中创建表结构（参考文档表3-10～3-13）。同样包含50名学生、10门课程、每生5门选课。
- GUI开发：功能与学院A/B一致，确保满足需求中的登录、选课、退选、查看共享课程等。
- 本地业务逻辑：处理本地选课及跨院选课逻辑。
- XML处理：使用DOM/SAX完成XML生成和解析。
- 通信模块：与集成服务器通信，实现数据交换。
交付物：
- 学院C完整源代码（MySQL脚本、GUI代码、XML处理、通信）
- 初始数据SQL文件
- 部署说明
1. 成员5：集成服务器核心开发
- 通信服务端：搭建轻量级HTTP服务器（如Spring Boot、Jetty或Netty），接收三个学院发来的请求，并路由到对应处理器。
- XML验证：基于Xerces或Java内置SchemaFactory，使用预定义的XSD验证收到的XML结构合法性。
- XML转换（XSLT）：
  - 实现从学院A/B/C本地XML格式到统一XML格式的转换（编写XSLT文件，如studentToUnify.xsl等）。
  - 实现从统一XML格式到目标学院本地XML格式的转换（如unifyToA.xsl等）。
- 课程共享处理：
  - 接收某学院请求“获取其他学院共享课程”，向其他两个学院转发请求，收集课程XML，转换为统一格式，合并后转换为请求学院格式并返回。
- 选课转发：
  - 接收来自学院A的跨院选课请求（包含学生信息和目标课程信息），验证后转发给目标学院B或C的教务系统，并将选课结果返回原学院。
- 退选课转发：类似选课转发，支持退选请求的转发与确认。
交付物：
- 集成服务器完整源代码
- 所有XSLT样式文件（至少9个：3个学院→统一，统一→3个学院，共6个，外加选课、退选相关）
- XSD验证文件
- 服务器部署与接口文档
1. 成员6：集成服务器扩展功能 & 联调
- 全局统计模块：
  - 实现集成服务器端统计功能：分别统计三个学院的学生总数、课程总数、选课总数（可通过实时查询各学院或定期汇总）。
  - 提供统计接口，供管理员或GUI调用，返回XML格式的统计结果（例如<statistics><college name="A"><students>50</students>...</college></statistics>）。
  - 可选：实现可视化展示（如饼图、柱状图）。
- 退选课完整流程：
  - 与成员5协作，实现退选请求的处理逻辑：学生在本院系统发起退选（跨院课程）→ 本院系统发送退选XML到集成服务器 → 集成服务器验证并转发至课程所属学院 → 目标学院删除选课记录 → 返回成功 → 本院系统更新本地课表。
  - 确保退选后全局统计数据正确更新。
- 系统联调与测试：
  - 搭建集成测试环境，模拟三个学院同时运行，验证课程共享、跨院选课、退选课、全局统计等端到端流程。
  - 编写测试用例（正常选课、重复选课、退选不存在课程等异常场景）。
  - 协助修复集成过程中的通信、XML格式、编码等问题。
- 性能与异常处理：增加日志记录、超时重试机制，保证系统健壮性。
交付物：
- 全局统计模块代码及接口说明
- 退选课流程实现代码
- 测试报告（包含测试用例、执行结果、问题记录）
- 联调环境配置指南

五、技术栈建议
- 语言：Java（统一语言便于集成）或允许各学院使用Python（但XML处理需统一）
- XML处理：DOM4J、JAXB、Xerces、Saxon（XSLT 2.0/3.0支持）
- 通信：HTTP + RESTful风格，数据格式XML（Content-Type: application/xml）
- 数据库：SQL Server、Oracle、MySQL（按学院分配）
- GUI：Java Swing/JavaFX 或 Python Tkinter/PyQt
- 构建工具：Maven/Gradle（管理依赖）
六、验收标准
1. 每个学院系统可独立登录、查看本校课程及共享课程。
2. 学院A的学生能够成功选修学院B的共享课程，并在学院B的数据库中生成选课记录。
3. 集成服务器能正确返回全局统计信息（学生数、课程数、选课数）。
4. 学生可退选已选的跨院课程，退选后双方数据库记录删除。
5. 所有跨系统数据交换均基于XML格式，并经过XSD验证和XSLT转换。
6. 报告包含系统架构图、课程共享流程图、跨院选课流程图、退选课流程图。
以上分工明确，覆盖了所有需求，并考虑到了6人小组的协作效率。每位成员均有独立可交付的模块，同时需要频繁沟通接口细节，确保集成成功。
